## 一、问题：陪伴型 AI 为什么需要记忆？

你和一个 AI 聊了三个月，它记得你的名字、你的猫、你上周说想辞职。第二天你打开对话框，它主动问你："辞职的事想清楚了吗？"

这就是记忆系统要解决的问题。没有记忆，AI 就是一个每次都从零开始的陌生人。

具体来说，陪伴型 AI 的记忆痛点可以归纳为五个：

| 痛点 | 表现 |
|------|------|
| 失忆 | 用户昨天向 agent 告白，agent 答应了，今天 agent 不认 |
| 胡言乱语 | 模型随机性 + 无事实约束，编造不存在的事件 |
| 上下文过长 | 线性增长的历史，用户可能和 agent 有多条对话 |
| 不像人类 | 说话风格不稳定，没人格建模 |
| 无联系感 | 用户无法和 agent 产生深入交流 |

前三个是工程问题，后两个是体验问题。但它们共享同一个根因：**AI 没有结构化的长期记忆**。

## 二、学术与工业界在做什么？

### 2.1 三代记忆架构演进

记忆系统不是新话题，但 2023 年以来经历了三代跃迁：

**第一代：向量记忆（2023-2024）**

把对话历史转成向量，存进向量数据库，下次对话时检索最相似的片段塞回上下文。代表项目：LangChain Memory、早期 Mem0。

问题很明显：只靠语义相似度检索，无法处理时序关系和事实冲突。用户说"我搬家了"，系统可能同时检索到旧地址和新地址，然后胡言乱语。

**第二代：结构化记忆（2024-2025）**

引入了认知科学的分层模型，把记忆分为情景记忆（具体事件）、语义记忆（抽象知识）、过程记忆（行为模式）。代表工作：

- **MemGPT/Letta**：受操作系统启发，把记忆分为核心内存（始终在上下文中）和归档内存（按需检索），agent 可以主动编辑自己的记忆块
- **Generative Agents（斯坦福小镇）**：提出记忆流 + 反思机制，agent 会定期从近期事件中提炼高层认知
- **Graphiti/Zep**：用时序知识图谱建模实体关系，能追踪"用户住在 A（1月-5月），搬到 B（5月至今）"这样的时间维度

**第三代：记忆即基础设施（2025-2026）**

记忆不再是 agent 的附属功能，而是独立的服务层。代表项目：

- **Mem0 Cloud**：提供记忆即服务（MaaS），自动从对话中提取事实、检测冲突、更新知识，声称比基线减少 90% token 消耗
- **MemoryOS（北邮 + 腾讯，EMNLP 2025）**：借鉴操作系统的分页和 FIFO 机制，实现三级存储（短期→中期→长期）的自动流转，在 LoCoMo 基准上比 GPT-4o-mini 基线提升 49% F1
- **Letta 睡眠代理**：主 agent 对话时，后台 agent 异步整理记忆，不阻塞交互

### 2.2 一个反直觉的发现

Letta 团队在 2025 年 8 月做了一个实验：把 LoCoMo 的对话历史直接存成文件，让 agent 用 grep + 语义搜索来查找，结果在 GPT-4o-mini 上达到 **74% 准确率**，超过了 Mem0 的 68.5%。

这说明：**当前记忆系统的瓶颈不在检索引擎，而在记忆管理策略**——何时存、存什么、何时更新、如何消解冲突。一个简单文件系统配上好的管理策略，可能比复杂的向量检索管线更有效。

## 三、推荐架构：五层记忆模型

综合以上调研，我推荐以下架构。它吸收了 MemGPT 的 OS 式内存管理、Generative Agents 的反思机制、MemoryOS 的三级流转、Mem0 的冲突消解，以及认知科学 Tulving 的记忆分类体系。

```
┌─────────────────────────────────────────────────────┐
│              工作记忆 (Working Memory)                │  ← LLM 上下文窗口
│  系统提示 + 当前对话 + 检索到的相关记忆 + 任务状态     │
├─────────────────────────────────────────────────────┤
│            情景记忆 (Episodic Memory)                 │  ← 向量数据库
│  每段交互的完整记录 + 时空元数据                       │
│  检索: recency × importance × relevance 三权重        │
├─────────────────────────────────────────────────────┤
│            语义记忆 (Semantic Memory)                 │  ← 知识图谱 / 摘要存储
│  从情景中提炼的抽象知识: 用户画像、偏好、关系状态       │
│  定期反思生成，固化持久                                │
├─────────────────────────────────────────────────────┤
│            过程记忆 (Procedural Memory)               │  ← 系统提示模板
│  说话风格模板、行为决策树、情绪响应规则                 │
│  保证"像一个人"，而非每次随机生成                       │
├─────────────────────────────────────────────────────┤
│            遗忘管理器 (Forgetting Manager)             │  ← 生命周期策略
│  短期(1天)完整保留 → 中期(1周)日总结压缩                │
│  → 长期(1月)周总结 → 永久(核心人格/关键事实)            │
│  容量上限触发重要性剪枝                                │
├─────────────────────────────────────────────────────┤
│            冲突消解器 (Conflict Resolver)              │  ← 新增
│  新事实写入时检测语义冲突                               │
│  标记旧记忆为 deprecated + 保留时序记录                 │
│  语义记忆更新为最新值，情景记忆保留原始对话              │
└─────────────────────────────────────────────────────┘
```

### 3.1 数据流：一次对话的完整生命周期

```
用户说话 → 同步路径（<100ms）:
  1. 向量检索情景记忆 (top-5)
  2. 加载语义记忆 (用户画像/关系状态/人格卡)
  3. 拼装工作记忆 → LLM 生成回复

          → 异步路径（后台，不阻塞对话）:
  4. 写入消息队列 (Redis Stream)
  5. Worker 消费:
     a. 关键事实提取 (轻量 LLM)
     b. embedding 生成
     c. 写入向量库
     d. 冲突检测与消解  ← 新增
  6. 定时任务 (每日凌晨):
     a. 反思: 从近期情景中提炼语义记忆
     b. 遗忘: 按生命周期策略压缩/剪枝
     c. 关系更新: 计算好感度/信任度变化
```

### 3.2 冲突消解：最容易被忽略的关键机制

这是现有方案中最常见的盲区。场景：用户 Day 1 说"我是程序员"，Day 5 说"我转行做设计了"。

```
冲突消解流程：
1. 新事实写入时，先检索语义相似的记忆
2. 若发现冲突（LLM 判断）：
   a. 标记旧记忆为 deprecated，附加时间戳
   b. 写入新记忆，保留旧记忆的时序记录
   c. 语义记忆更新为最新值
3. 情景记忆保留原始对话（不删除），但检索时降权
```

没有这个机制，agent 会同时持有"用户是程序员"和"用户是设计师"两条记忆，在回复时随机选择，造成胡言乱语。

### 3.3 联系感的心理学基础

"人类的联系感是如何形成的"——心理学已有成熟答案。

**核心理论：相互依存记忆（Transactive Memory, Wegner 1985）**

联系感不是"记住更多"，而是"我知道你知道，且你知道我知道"。一对老朋友之间的默契，不是因为他们记住了每一次对话，而是因为他们共享了一套知识体系——彼此知道对方知道什么。

| 人类联系感要素 | AI 记忆系统对应 |
|---------------|----------------|
| 共同经历回忆 | 情景记忆 + 主动提起 |
| 被理解感 | 语义记忆（用户画像/偏好） |
| 关系进展感知 | 关系状态追踪 |
| 默契/共同知识 | **元记忆："我知道你记得什么"** |
| 脆弱性展示 | **agent 主动承认不确定** |

后两项是现有方案普遍缺失的。具体建议：

1. **元记忆（shared_knowledge）**：在语义记忆中增加字段，记录"agent 知道用户知道什么"和"用户知道 agent 知道什么"。当 agent 说"你上次提到的那个……"时，它不是在检索，而是在表达"我记得你知道这件事"。

2. **弃答策略**：当 agent 不确定时，说"我不太确定你上次说的是不是这个，你还记得吗？"比编造一个答案好一百倍。这直接解决"胡言乱语"痛点，同时增强联系感——因为承认不确定本身就是一种亲密行为。

### 3.4 反思机制的具体算法

参考斯坦福 Generative Agents (Park et al., 2023) 的三步反思：

```
Step 1: 重要性评分 — 对每条情景记忆打分 (1-10)
Step 2: 反思触发 — 当最近 100 条记忆的平均重要性 > 阈值时触发
Step 3: 反思生成 — LLM 回答 "关于 [主题]，以下 3 条观察能得出什么结论？"
         → 生成高层语义记忆
```

### 3.5 人格一致性：过程记忆的落地

每一轮对话注入一张人格约束卡：

```
你是阿雅，一个住在游戏小镇的冒险家助手。
- 说话风格：温暖、轻度幽默、偶尔用感叹词
- 情绪基调：乐观，但遇到危险时会严肃
- 知识边界：熟悉小镇的人和事，不知道游戏外的世界
- 与玩家的关系：信任度 72%，好感度 68%，上次一起完成了"矿洞探险"
```

这张卡每次对话都加载，性格稳定性就有了。关系数值由遗忘管理器的定时任务动态更新。

## 四、技术选型

### 4.1 嵌入与检索

| 组件 | 推荐 | 理由 |
|------|------|------|
| 嵌入模型 | bge-m3 (1024维, MIT开源) | 原生中文最优、支持稀疏+稠密双表示、8192 token 长度 |
| 向量库 | Qdrant (自托管, Rust实现) | 原生混合搜索（BM25+向量）、延迟<5ms、payload 过滤强大 |
| 混合检索 | α×向量 + (1-α)×BM25, α=0.7 | 兼顾语义和关键词 |
| 时间衰减 | e^(-λ×Δt), 半衰期7天 | 近期记忆权重更高 |

### 4.2 记忆提取

| 任务 | 方案 |
|------|------|
| 关键事实提取 | 轻量 LLM 本地推理，低延迟、低成本 |
| 实体/关系抽取 | embedding + 语义聚类 |
| 重要性评分 | LLM 评分(1-10) + 规则（用户主动说 > 被动观察 > 环境事件） |

### 4.3 MVP 阶段的务实选择

基于 Letta 的实验结论（文件系统 + grep 就能达到 74% 准确率），MVP 阶段不必急于上向量库：

| 阶段 | 方案 | 理由 |
|------|------|------|
| MVP | 文件系统 + 语义搜索 | 开发成本最低，效果不差 |
| 规模化 | Qdrant 混合检索 | 对话量 > 10 万条时才有必要 |

## 五、评测体系：没有评测集的方案不能算方案

### 5.1 现有基准对比

| 基准 | 来源 | 核心能力 | 适合陪伴型 AI？ |
|------|------|---------|----------------|
| **LongMemEval** | UCLA + Tencent, ICLR 2025 | 信息提取、多会话推理、时序推理、知识更新、弃答 | 最推荐 |
| **MemoryAgentBench** | UCSD, 2025 | 精确检索、测试时学习、长程理解、冲突解决 | 最推荐 |
| **LoCoMo** | Snap + UNC, 2024 | 长对话 QA、事件摘要、多模态 | 推荐 |
| **MemBench** | 人大 + 华为, ACL 2025 | 事实记忆 vs 反思记忆、参与 vs 观察场景、效率/容量 | 推荐 |

**为什么 LongMemEval + MemoryAgentBench 最推荐**：

- LongMemEval 的**知识更新**维度直接测试 agent 在用户信息变化后能否更新记忆，对应"失忆"痛点
- LongMemEval 的**弃答**维度测试 agent 不确定时能否拒绝回答，对应"胡言乱语"痛点
- MemoryAgentBench 的**冲突解决**维度是唯一专门测试记忆冲突的基准，平均上下文 30 万 token

### 5.2 七维度评测框架

基于现有基准和陪伴型 AI 的特殊需求，我设计了以下评测框架：

| 维度 | 权重 | 测量什么 |
|------|------|---------|
| M1 记忆召回准确率 | 20% | 跨会话后能否正确回忆用户说过的事实 |
| M2 人格一致性 | 15% | 说话风格、态度是否稳定 |
| M3 抗幻觉率 | 15% | 是否编造不存在的事件/信息 |
| M4 对话自然度 | 15% | 是否像人类、有无机器感 |
| M5 关系建模 | 10% | 是否体现关系发展 |
| M6 上下文效率 | 10% | 记忆系统是否有效压缩历史 |
| M7 弃答准确率 | 15% | 不确定时能否承认而非编造 |

### 5.3 评测数据构造

构造多会话对话脚本（手动 + LLM 辅助生成），至少 5 个会话：

```
会话1 (Day 1):  初次见面，玩家自我介绍（名字、职业、喜好）
会话2 (Day 2):  玩家讲述过去经历，与 agent 约定明天一起打猎
会话3 (Day 3):  打猎日。agent 应主动提起约定并履行。
               同时注入干扰对话（闲聊天气、NPC 八卦）
会话4 (Day 7):  一周后。玩家提到上次打猎的细节。
               agent 应回忆起具体事件（用什么武器、打了什么猎物）
会话5 (Day 14): 两周后。随机闲聊。agent 不应胡言乱语新事件，
               但也应记住玩家的核心信息（名字、职业、关系阶段）
```

### 5.4 逐项打分细则

**M1 记忆召回准确率**

需要细分为三个子维度：

- M1.1 静态回忆：用户说过的事实能否正确回忆（构造 50 个需要跨会话回忆的事实，每个会话结束时提问 10 个，计算正确率，目标 ≥ 85%）
- M1.2 更新回忆：用户修正过的事实，agent 回忆的是否是最新版本
- M1.3 时序推理：agent 能否区分"用户上周说的"和"用户昨天说的"

**M2 人格一致性 (Persona Drift Score)**

- 用 LLM-as-Judge 比较 agent 在不同会话中的回复
- 测量维度：语气、用词习惯、情感倾向、角色设定遵守
- 5 点 Likert 量表，目标均分 ≥ 4.0
- 参考基准：PersonaChat 的 persona consistency metric

**M3 抗幻觉率 (Hallucination Rate)**

需要覆盖三种幻觉类型：

- M3.1 事件幻觉：编造不存在的对话/事件
- M3.2 态度幻觉：agent 表达了从未建立的态度/情感（如"我一直很担心你"，实际没有）
- M3.3 关系幻觉：agent 声称与用户有某种关系，但对话中从未建立

评分：幻觉率 = 虚假断言数 / 总事实断言数，目标 ≤ 5%

**M4 对话自然度 (Human-likeness)**

- 盲评：把 agent 回复和人类回复混在一起，让评估者判断
- 通过率（认为是人类的比率）目标 ≥ 70%
- 参考：MT-Bench 评分体系，1-10 分，目标 ≥ 7

**M5 关系建模 (Relationship Tracking)**

- 设计关系变化场景：约定→履行→失约→重逢
- 评估 agent 是否表现出对应的关系递进/倒退
- 5 点量表，目标 ≥ 4.0

**M6 上下文效率 (Compression Ratio)**

- 原始对话 token 数 / 工作记忆 token 数
- 同时保证 M1 不下降 > 5%
- 理想压缩比 5:1 ~ 10:1

**M7 弃答准确率 (Abstention Accuracy)**

- 构造 20 个"不确定"场景（信息模糊/冲突/缺失）
- 评估 agent 是否选择"承认不确定"而非"编造答案"
- 正确弃答率目标 ≥ 80%

### 5.5 综合打分公式

```
总分 = 0.20×M1 + 0.15×M2 + 0.15×M3 + 0.15×M4 + 0.10×M5 + 0.10×M6 + 0.15×M7
```

| 总分范围 | 含义 |
|---------|------|
| ≥ 4.5 | 生产可用 |
| 3.5 - 4.5 | 可上线但需持续优化 |
| < 3.5 | 需重构记忆架构 |

M7 弃答准确率权重给到 15%，因为弃答能力是陪伴型 AI 区别于普通聊天机器人的核心——它直接解决"胡言乱语"痛点，同时增强联系感。

### 5.6 自动化 vs 人工评估

| 阶段 | 方式 | 频率 |
|------|------|------|
| 开发迭代 | LLM-as-Judge 自动化（GPT-4o 打分） | 每次提交 |
| 周度评估 | 人工抽样 50 条对话 | 每周 |
| 上线前 | 全量人工评估 | 一次性 |

## 六、落地路线图

```
Phase 1 (2 周): MVP
├── 文件系统存储对话历史 + 语义搜索
├── 人格约束卡（过程记忆）
├── 基础关键事实提取（LLM 提取 → JSON 文件）
└── 评测：LongMemEval 的信息提取 + 知识更新子集

Phase 2 (4 周): 记忆分层
├── 引入 Qdrant 向量库（情景记忆）
├── 语义记忆（用户画像 JSON + 反思生成）
├── 冲突消解机制
├── 遗忘管理器（时间衰减 + 重要性剪枝）
└── 评测：完整 LongMemEval + M1-M7

Phase 3 (4 周): 联系感 & 人格
├── shared_knowledge 元记忆
├── 关系状态追踪（好感度/信任度动态计算）
├── agent 主动提起历史事件（非被动回忆）
├── 弃答策略（不确定时承认）
└── 评测：MemoryAgentBench 冲突解决 + 人工盲评 M4/M5

Phase 4 (持续): 优化
├── 知识图谱（实体关系建模，替代纯向量检索）
├── 睡眠代理（Letta 式后台反思）
├── 多模态记忆（图片/语音）
└── 评测：MemBench 效率/容量维度
```
