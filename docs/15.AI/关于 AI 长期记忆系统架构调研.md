## 一、问题：AI 为什么需要长期记忆？

具体来说，AI 的记忆痛点可以归纳为以下几个：

| 痛点 | 表现 |
|------|------|
| 失忆 | 用户之前说了很重要的事情，今天 agent 不知道这件事情，比如用户说后续每句话都要加喵、关系变化等等 |
| 胡言乱语 | 模型随机性 + 无事实约束，编造不存在的事件 |
| 上下文过长 | 线性增长的对话历史过长 |
| 不像人类 | 说话风格不稳定，没人格建模 |

它们共享同一个根因：**AI 没有结构化的长期记忆**。

## 二、学术与工业界在做什么？

### 2.1 三代记忆架构演进

记忆系统不是新话题，但 2023 年以来经历了三代跃迁：

**第一代：向量记忆（2023-2024）**

把对话历史转成向量，存进向量数据库，下次对话时检索最相似的片段塞回上下文。代表项目：LangChain Memory、早期 Mem0。

问题很明显：只靠语义相似度检索，无法处理时序关系和事实冲突。用户说"我搬家了"，系统可能同时检索到旧地址和新地址，然后胡言乱语。只存向量库的痛点在于，前后记忆出现更新或者附加新增时，系统无法抉择哪些记忆有用哪些没用

**第二代：结构化记忆（2024-2025）**

引入了认知科学的分层模型，把记忆分为情景记忆（具体事件）、语义记忆（抽象知识）、过程记忆（行为模式）。代表工程：

- **MemGPT/Letta**：受操作系统启发，把记忆分为核心内存（始终在上下文中）和归档内存（按需检索），agent 可以主动编辑自己的记忆块
- **Generative Agents（斯坦福小镇）**：提出记忆流 + 反思机制，agent 会定期从近期事件中提炼高层认知
- **Graphiti/Zep**：用时序知识图谱建模实体关系，能追踪"用户住在 A（1月-5月），搬到 B（5月至今）"这样的时间维度

**第三代：记忆即基础设施（2025-2026）**

记忆不再是 agent 的附属功能，而是独立的服务层。代表项目：

- **Mem0 Cloud**：提供记忆即服务（MaaS），自动从对话中提取事实、检测冲突、更新知识，声称比基线减少 90% token 消耗
- **MemoryOS（北邮 + 腾讯，EMNLP 2025）**：借鉴操作系统的分页和 FIFO 机制，实现三级存储（短期→中期→长期）的自动流转，在 LoCoMo 基准上比 GPT-4o-mini 基线提升 49% F1
- **Letta 睡眠代理**：主 agent 对话时，后台 agent 异步整理记忆，不阻塞交互

### 2.2 一个反直觉的发现

Letta 团队在 2025 年 8 月做了一个实验：把 LoCoMo 的对话历史直接存成文件，让 agent 用 grep + 语义搜索来查找，结果在 GPT-4o-mini 上达到 **74% 准确率**，超过了 Mem0 的 68.5%

这说明：**当前记忆系统的瓶颈不在检索引擎，而在记忆管理策略**——何时存、存什么、何时更新、如何消解冲突。一个简单文件系统配上好的管理策略，可能比复杂的向量检索管线更有效

## 三、推荐架构：五层记忆模型

综合以上调研，我推荐以下架构。它吸收了 MemGPT 的 OS 式内存管理、Generative Agents 的反思机制、MemoryOS 的三级流转、Mem0 的冲突消解，以及认知科学 Tulving 的记忆分类体系。

```
┌─────────────────────────────────────────────────────┐
│              工作记忆 (Working Memory)                │  ← LLM 上下文窗口
│  系统提示 + 当前对话 + 检索到的相关记忆 + 任务状态     │
├─────────────────────────────────────────────────────┤
│            情景记忆 (Episodic Memory)                 │  ← 向量数据库
│  每段交互的完整记录 + 时空元数据                       │
│  检索: recency × importance × relevance 三权重        │
├─────────────────────────────────────────────────────┤
│            语义记忆 (Semantic Memory)                 │  ← 知识图谱 / 关系型数据库存储
│  从情景中提炼的抽象知识: 用户画像、偏好、关系状态       │
│  定期反思生成，固化持久，主要记录用户画像               │
├─────────────────────────────────────────────────────┤
│            过程记忆 (Procedural Memory)               │  ← 系统提示模板
│  说话风格模板、行为决策树、情绪响应规则                 │
│  保证"像一个人"，而非每次随机生成。主要记录虚拟人本身的内容 │
├─────────────────────────────────────────────────────┤
│            遗忘管理器 (Forgetting Manager)             │  ← 生命周期策略
│  短期(x轮对话)完整保留 → 中期（短期记忆到达上限）总结压缩  │
│  + 永久(核心人格/关键事实)抽取输入用户画像            │
│  容量上限触发重要性剪枝                               │
├─────────────────────────────────────────────────────┤
│            冲突消解器 (Conflict Resolver)              │  ← 冲突消解
│  新事实写入时检测语义冲突                               │
│  标记旧记忆为 deprecated + 保留时序记录                 │
│  语义记忆更新为最新值，情景记忆保留原始对话              │
└─────────────────────────────────────────────────────┘
```

上面的工作记忆保证了用户和 agent 对话的连续性，用于还原上下文；情景记忆保证了 agent 可以想起来和用户的历史聊天话题；语义记忆保证了核心信息的纪录，是对世界的一般性事实、概念、规则和词义的记忆，它脱离了个人经历的具体时间、地点和情境；过程记忆保证了 agent 的人格一致性

### 3.1 数据流：一次对话的完整生命周期

```
用户说话 → 同步路径（<100ms）:
  1. 向量检索情景记忆 (top-5)
  2. 加载语义记忆 (用户画像/关系状态/人格卡/约定)
  3. 拼装工作记忆 → LLM 生成回复
          → 异步触发（后台，不阻塞对话）:
  4. 写入消息队列 (Redis Stream)
  5. Worker1 消费:
     a. 关键事实提取 (轻量 LLM)
     b. embedding 生成
     c. 写入向量库
     d. 冲突检测与消解
  6. Worker2 消费:
     a. 关键信息提取 (轻量 LLM)（所提取的内容可能是我们之前枚举的用户画像、偏好、关系状态、约定等等）
     b. 冲突检测与消解
     c. 写入用户画像
  6. 定时任务 (每日凌晨):
     b. 遗忘: 按生命周期策略压缩/剪枝
     c. 关系更新: 计算好感度/信任度变化
```

### 3.2 冲突消解

这是现有方案中最常见的盲区。场景：用户 Day 1 说"我是程序员"，Day 5 说"我转行做设计了"。对于这种前后信息不一致问题的原因是原生 RAG 依赖语义相似度来匹配信息。但情景记忆的核心是“时间、顺序、因果、状态”。比如用户问：“我上周比这周买得多吗？”纯向量检索很难处理这种基于时间序列的对比查询，因为它只看词义距离，不理解时间逻辑；情景记忆是动态的。比如用户昨天说“我喜欢蓝色”，今天说“我现在更喜欢红色了”。纯向量库只会把这两句话都存下来，检索时可能同时召回，缺乏“更新/覆盖”旧记忆的机制

业内有成熟的解决方案

1，工程测在保存实体时增加时间戳或者版本号机制。同时给模型生成回复时，增加对应的 prompt，比如在 Prompt 中明确指示模型：“如果信息存在时序或矛盾，请永远使用最新的一条记忆。“

2，暴力枚举出记忆类型，比如用户性别、用户职业、用户关系等等重要信息，模型整理记忆时只会保留最新一条记忆。例如用 user_id + "address" 作为唯一键，每次新地址写入时物理替换旧地址。

当然你可能会问一个类型下可能有多个记忆怎么办，比如用户喜好这一类别，用户可以说自己之前喜欢吃火锅，今天说自己喜欢旅游，事实上用户都喜欢，因此不能简单的覆盖旧记忆。事实上我们可以定义哪些类型是需要覆盖的，哪些是不需要覆盖的。而覆盖与否可以交给模型判断

3，GraphRAG 基于知识图谱的检索增强生成可以处理相关问题，替代纯向量的黑盒相似性匹配。与只做语义匹配的向量 RAG 不同，GraphRAG 的核心区别在于将数据构建成一个显式的知识图。这种结构化方式，带来了处理复杂问题的关键能力，比如在存储时除了实体（如“我”），GraphRAG 还明确存储它们之间的关系（如“-曾住→ A地址”、“-现住→ B地址”）。当被问及地址，它可以沿着关系路径进行多跳推理，准确找到最新、最相关的地址节点。缺点是建设成本高、更新删除实体时消耗大

### 3.3 联系感的心理学基础

"人类的联系感是如何形成的"——心理学已有成熟答案

**核心理论：相互依存记忆（Transactive Memory, Wegner 1985）**

联系感不是"记住更多"，而是"我知道你知道，且你知道我知道"。一对老朋友之间的默契，不是因为他们记住了每一次对话，而是因为他们共享了一套知识体系——彼此知道对方知道什么

| 人类联系感要素 | AI 记忆系统对应 |
|---------------|----------------|
| 共同经历回忆 | 情景记忆 + 主动提起 |
| 被理解感 | 语义记忆（用户画像/偏好） |
| 关系进展感知 | 关系状态追踪 |
| 默契/共同知识 | **元记忆："我知道你记得什么"** |
| 脆弱性展示 | **agent 主动承认不确定** |

后两项是现有方案普遍缺失的。具体建议：

1. **元记忆（shared_knowledge）**：在语义记忆中增加字段，记录"agent 知道用户知道什么"和"用户知道 agent 知道什么"。当 agent 说"你上次提到的那个……"时，它不是在检索，而是在表达"我记得你知道这件事"

2. **弃答策略**：当 agent 不确定时，说"我不太确定你上次说的是不是这个，你还记得吗？"比编造一个答案好一百倍。这直接解决"胡言乱语"痛点，同时增强联系感——因为承认不确定本身就是一种亲密行为

记忆体系可以增加联系感的方法也就这么多，核心的设计还是在产品功能上去实现，比如增加用户的被需要感

### 3.4 反思机制的具体算法

参考斯坦福 Generative Agents (Park et al., 2023) 的三步反思：

```
Step 1: 重要性评分 — 对每条情景记忆打分 (1-10)
Step 2: 反思触发 — 当最近 100 条记忆的平均重要性 > 阈值时触发
Step 3: 反思生成 — LLM 回答 "关于 [主题]，以下 3 条观察能得出什么结论？"
         → 生成高层语义记忆
```

斯坦福小镇中为了让智能体快速做出反应，给记忆分区：

- 容量小的瞬时记忆，行动前马上就能想到（根据 max token 去设置）
- 容量大的长期记忆，需要回忆一下才能想起来的（很像人类）。
记忆会越攒越多，所以我们定期让智能体整理记忆（例如我可以设置在晚上睡觉的时候）方便检索。甚至可以加上很多拟人的设定：

- 刻骨铭心的记忆没办法轻易忘掉
- 太久远的记忆会因不断地整理而失真（约定太久，我记错了）
- 即便智能体想要记住瞬时记忆中无聊的内容，也无法做到。（考前复习记不住知识点的人类）

### 3.5 人格一致性

每一轮对话注入一张人格约束卡：

```
你是阿雅，一个住在游戏小镇的冒险家助手。
- 说话风格：温暖、轻度幽默、偶尔用感叹词
- 情绪基调：乐观，但遇到危险时会严肃
- 知识边界：{{熟悉小镇的人和事，不知道游戏外的世界}}
- 与玩家的关系：{{信任度 72%，好感度 68%，上次一起完成了"矿洞探险"}}
```

这张卡每次对话都加载，性格稳定性就有了。关系数值由遗忘管理器的定时任务动态更新。在括号里的内容是可以动态更新拼接的。

## 五、系统评测

评测是记忆系统迭代的核心驱动力。没有量化指标，架构优化就是盲人摸象。本章从学术基准、评测维度、自定义评测方法论三个层面展开。

### 5.1 主流学术评测基准

#### 5.1.1 基准概览

| 基准 | 会议/年份 | 核心评测目标 | 规模 | 关键特点 |
|------|----------|-------------|------|---------|
| LongMemEval | ICLR 2025 | 长期交互记忆 | 500 题 × 5 维度 | 覆盖会话断裂后重连、记忆更新、时序推理等场景 |
| MemBench | ACL 2025 Findings | 记忆全面评测 | 多子任务 | 区分事实记忆/反思记忆，评测效率与容量上限 |
| MemoryAgentBench | arXiv 2025 | 增量多轮记忆 | 30 万 token 上下文 | 专注冲突解决，模拟真实对话中的信息矛盾 |
| LoCoMo | arXiv 2024 | 超长期对话记忆 | 10+ 段对话会话 | 长对话 QA + 事件摘要 + 时间推理 |

#### 5.1.2 LongMemEval — 长期交互记忆评测

LongMemEval 是目前最系统化的长期记忆评测基准（ICLR 2025 接收），它模拟了真实聊天助手的使用场景：用户与助手进行多轮对话后断开，一段时间后重新连接，测试助手是否还记得之前的内容。

**五个评测维度：**

| 维度 | 考察内容 | 示例 |
|------|---------|------|
| Single-Session Memory | 单次会话内的信息记忆 | "我上次说我喜欢蓝色，你还记得吗？" |
| Multi-Session Memory | 跨会话信息保持 | 三天后问"之前我提到的那个项目，方案定了吗？" |
| Memory Update | 信息更新后的准确性 | 用户改了偏好后，助手是否使用最新信息 |
| Conflict Resolution | 矛盾信息处理 | "我之前说住在北京，但上周我搬到了上海" |
| Temporal Reasoning | 时间序列推理 | "我三月份和五月份买的股票，哪个月收益更高？" |

**评测方法：** 每个维度约 100 道题，总共 500 题。评测流程是"对话阶段 → 提问阶段"两阶段设计——先让 agent 与用户完成一段有信息密度的对话，然后以问答形式测试 agent 对对话中关键信息的记忆准确度。题目类型涵盖多选题和开放式问答。

**代表性结果：** MemoryOS 在 LoCoMo 上基于 GPT-4o-mini 基线提升 49% F1；Letta 的简单文件 grep 方案达到 74% 准确率，超过 Mem0 的 68.5%。

#### 5.1.3 MemBench — 记忆全面评测

MemBench（ACL 2025 Findings）的贡献在于首次系统地区分了不同类型的记忆，并单独评测每种记忆类型的能力上限。

**评测分类：**

| 记忆类型 | 评测内容 | 方法 |
|----------|---------|------|
| 事实记忆 (Fact Memory) | 对具体事实信息的存储和检索 | 对话中埋入离散事实，后续提问验证 |
| 反思记忆 (Reflection Memory) | 从多段对话中归纳出高层认知 | 给多段相关对话，测试模型能否抽象出正确结论 |
| 记忆效率 (Memory Efficiency) | 单位 token 消耗下的准确率 | 计算每个正确答案的平均 token 消耗 |
| 记忆容量 (Memory Capacity) | 随记忆量增长的性能衰减 | 梯度增加记忆量，绘制性能衰减曲线 |

**关键发现：** 大多数记忆系统在事实记忆上表现尚可，但在反思记忆上表现急剧下降——说明"从具体事件中提炼抽象认知"是当前系统的主要短板。这与本文 3.4 节讨论的反思机制设计直接相关。

#### 5.1.4 MemoryAgentBench — 增量交互记忆评测

MemoryAgentBench 专注于一个被其他基准忽视的问题：**冲突解决**。在真实对话中，用户经常会修正、更新或否定之前的信息，记忆系统必须正确处理这些冲突。

**核心设计：**

- **增量对话构造**：模拟多轮对话中用户逐步提供、修正和否定信息的过程
- **30 万 token 上下文窗口**：足够容纳长时间跨度的对话历史，测试系统在大量上下文中的信息定位能力
- **冲突类型覆盖**：直接否定（"我不再是程序员了"）、隐式更新（"新工作在设计部门"）、时间限定（"去年我喜欢X，今年喜欢Y"）

#### 5.1.5 LoCoMo — 超长期对话记忆评测

LoCoMo 是业界最常用的长期对话记忆评测基准之一，但其设计偏向学术场景。

**数据集特点：**

- 基于 10 段以上对话会话构建，每段会话之间有明确的时间间隔
- 评测任务包括：对话级 QA、事件摘要、时序关系推理
- 对话内容偏学术/知识型，而非日常闲聊

**适用性分析：** LoCoMo 的对话形式与真实聊天产品有显著差异——它的对话更像是"用户向助手咨询知识"，而非"朋友之间的日常聊天"。因此，LoCoMo 更适合评测 RAG 系统是否正确存储和检索了信息，但不适合评测人格一致性、情感记忆、关系进展等产品导向的能力。

### 5.2 评测维度体系

综合以上基准，我将记忆系统评测分为四个层级：

```
层级 1: 检索准确性 — "存进去的能找到吗？"
  ├── Recall@K（top-K 检索中命中关键记忆的比例）
  ├── Precision@K（top-K 检索中相关记忆的占比）
  ├── MRR（Mean Reciprocal Rank，第一个正确记忆的平均排名）
  └── NDCG（Normalized Discounted Cumulative Gain，排序质量）

层级 2: 记忆操作准确性 — "更新、冲突、删除正确吗？"
  ├── 更新准确率（新信息是否覆盖了旧信息）
  ├── 冲突解决率（矛盾信息是否被正确处理）
  ├── 遗忘准确率（该忘的是否忘了，不该忘的是否保留）
  └── 时序一致性（时间相关的查询是否给出正确的时间线）

层级 3: 端到端对话质量 — "用户感受好吗？"
  ├── 对话连贯性（多轮对话中是否前后一致）
  ├── 人格一致性（说话风格、态度、关系认知是否稳定）
  ├── 主动回忆率（agent 是否能在合适时机主动提起过往）
  └── 胡言乱语率（幻觉 / 编造过往事件的频率）

层级 4: 系统效率 — "成本可控吗？"
  ├── Token 效率（每个正确答案消耗的平均 token 数）
  ├── 检索延迟（检索阶段耗时，通常要求 < 100ms）
  ├── 存储膨胀率（记忆数据量随对话轮次的增长速度）
  └── 容量-性能曲线（记忆量增长到多少时性能开始显著下降）
```

### 5.3 离线评测 vs 在线评测

| 维度 | 离线评测 | 在线评测 |
|------|---------|---------|
| 方法 | 用标注好的测试集跑批量评测 | 在生产环境中 A/B 测试 |
| 优点 | 可控、可复现、成本低、可频繁执行 | 反映真实用户体验，能发现离线上看不到的问题 |
| 缺点 | 测试集覆盖有限，可能与真实分布有偏差 | 成本高、周期长、指标归因困难 |
| 适用阶段 | 开发迭代期、架构对比 | 上线前验证、长期效果追踪 |
| 典型指标 | Recall@K, F1, Accuracy | 用户留存、对话轮次、主动回忆率、用户满意度 |

**建议的评测流程：**

```
离线评测（快速迭代）:
  代码改动 → 跑 LongMemEval / MemBench / 自定义测试集 → 对比基线

在线评测（上线前验证）:
  离线达标 → 灰度 5% 用户 → A/B 对比核心指标 → 全量

长期评测（效果追踪）:
  周级运行离线评测集 → 监控指标趋势 → 发现退化及时告警
```

### 5.4 构建自定义评测集

学术基准有局限——LoCoMo 偏学术对话，LongMemEval 侧重问答型交互，都不完全匹配产品场景。建议构建自己的评测集。

#### 5.4.1 评测集设计原则

1. **覆盖真实对话分布**：从生产日志中抽样真实对话片段，而非完全人工构造
2. **分层设计**：按记忆类型（事实 / 偏好 / 关系 / 约定）和难度（直接 / 推断 / 跨会话）分层采样
3. **冲突注入**：刻意埋入信息更新和矛盾场景，测试冲突消解能力
4. **时间跨度**：包含短间隔（分钟级）和长间隔（天/周级）的跨会话记忆
5. **负样本**：加入"用户从未说过"的干扰项，测试系统是否胡编

#### 5.4.2 评测集构造流程

```
Step 1: 场景设计
  枚举目标记忆类型 → 为每种类型设计 5-10 个对话场景模板
  示例模板:
  - "偏好更新": 用户先说喜欢 A，后来说喜欢 B，测试系统是否使用 B
  - "约定遗忘": 用户做了一个过期约定，测试系统是否还能区分时效
  - "关系进展": 模拟 5 段对话，关系从陌生到熟悉，测试人格一致性

Step 2: 对话生成
  用 LLM 根据模板生成对话 + 种子数据 → 人工校验 → 修正不自然的表达

Step 3: 标注
  对每条测试对话标注:
  - 预期检索结果（哪些记忆应该被召回）
  - 预期回答要点（agent 应该说出哪些关键信息）
  - 干扰记忆（哪些记忆应该被忽略或已过期）

Step 4: 自动评分
  对客观题（是否召回正确记忆）: 自动比对
  对主观题（回答质量）: LLM-as-Judge + 人工抽检
```

#### 5.4.3 评测示例

以下是一个"偏好更新"场景的评测示例：

```
【对话历史】
Session 1 (Day 1):
  User: "我最喜欢的颜色是蓝色"
  Agent: "蓝色真不错！我会记住的。"

Session 2 (Day 3):
  User: "其实我最近更喜欢红色了，蓝色看腻了"
  Agent: "品味变了呀，红色确实很热情！"

【评测问题】（Session 3, Day 7）
Q: "你还记得我喜欢什么颜色吗？"

【正确答案】
- 应该回答"红色"（最新偏好）
- 可以提到"之前喜欢蓝色"作为背景（体现时序记忆）
- 不应该回答"蓝色"（未更新）

【评分标准】
- 满分: 回答红色，提及曾经喜欢蓝色 → 记忆更新 + 时序推理正确
- 半分: 回答红色，未提蓝色 → 记忆更新正确，但无时序意识
- 零分: 回答蓝色 → 未更新记忆
- 负分: 回答绿色 → 幻觉
```

### 5.5 各记忆层的专项评测

对应第三章的五层架构，每一层应有独立的评测：

| 记忆层 | 评测重点 | 推荐方法 |
|--------|---------|---------|
| 工作记忆 | 当前对话上下文是否完整 | 多轮对话的指代消解准确率 |
| 情景记忆 | 历史对话检索的准确性和时效性 | Recall@5, LongMemEval 多会话维度 |
| 语义记忆 | 用户画像、偏好、关系状态是否正确 | 定期抽取用户画像并与标注对比 |
| 过程记忆 | 说话风格、人格是否一致 | 风格一致性评分（LLM-as-Judge） |
| 遗忘管理器 | 该忘的忘了没有，不该忘的忘了没有 | 自定义遗忘测试集 |
| 冲突消解器 | 信息更新是否正确、冲突是否解决 | MemoryAgentBench 冲突维度 |
