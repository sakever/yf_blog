关于 AI 长期记忆系统架构调研

# 一、问题：AI 为什么需要长期记忆？

具体来说，AI 的记忆痛点可以归纳为以下几个：

痛点 表现
失忆 用户之前说了很重要的事情，今天 agent 不知道这件事情，比如用户说后续每句话都要加喵、关系变化等等
胡言乱语 模型随机性 + 无事实约束，编造不存在的事件
上下文过长 线性增长的对话历史过长
不像人类 说话风格不稳定，没人格建模

它们共享同一个根因：AI 没有结构化的长期记忆。

# 二、学术与工业界在做什么？

# 2.1 三代记忆架构演进

记忆系统不是新话题，但 2023 年以来经历了三代跃迁：

第一代：向量记忆（2023-2024）

把对话历史转成向量，存进向量数据库，下次对话时检索最相似的片段塞回上下文。代表项目：LangChain Memory、早期 Mem0。

问题很明显：只靠语义相似度检索，无法处理时序关系和事实冲突。用户说"我搬家了"，系统可能同时检索到旧地址和新地址，然后胡言乱语。只存向量库的痛点在于，前后记忆出现更新或者附加新增时，系统无法抉择哪些记忆有用哪些没用

第二代：结构化记忆（2024-2025）

引入了认知科学的分层模型，把记忆分为情景记忆（具体事件）、语义记忆（抽象知识）、过程记忆（行为模式）。代表工程：

  • MemGPT/Letta：受操作系统启发，把记忆分为核心内存（始终在上下文中）和归档内存（按需检索），agent 可以主动编辑自己的记忆块
  • Generative Agents（斯坦福小镇）：提出记忆流 + 反思机制，agent 会定期从近期事件中提炼高层认知
  • Graphiti/Zep：用时序知识图谱建模实体关系，能追踪"用户住在 A（1月-5月），搬到 B（5月至今）"这样的时间维度

第三代：记忆即基础设施（2025-2026）

记忆不再是 agent 的附属功能，而是独立的服务层。代表项目：

  • Mem0 Cloud：提供记忆即服务（MaaS），自动从对话中提取事实、检测冲突、更新知识，声称比基线减少 90% token 消耗
  • MemoryOS（北邮 + 腾讯，EMNLP 2025）：借鉴操作系统的分页和 FIFO 机制，实现三级存储（短期→中期→长期）的自动流转，在 LoCoMo 基准上比 GPT-4o-mini 基线提升 49% F1
  • Letta 睡眠代理：主 agent 对话时，后台 agent 异步整理记忆，不阻塞交互

# 2.2 一个反直觉的发现

Letta 团队在 2025 年 8 月做了一个实验：把 LoCoMo 的对话历史直接存成文件，让 agent 用 grep + 语义搜索来查找，结果在 GPT-4o-mini 上达到 74% 准确率，超过了 Mem0 的 68.5%

这说明：当前记忆系统的瓶颈不在检索引擎，而在记忆管理策略——何时存、存什么、何时更新、如何消解冲突。一个简单文件系统配上好的管理策略，可能比复杂的向量检索管线更有效

# 三、推荐架构：五层记忆模型

综合以上调研，我推荐以下架构。它吸收了 MemGPT 的 OS 式内存管理、Generative Agents 的反思机制、MemoryOS 的三级流转、Mem0 的冲突消解，以及认知科学 Tulving 的记忆分类体系。

┌─────────────────────────────────────────────────────┐
│              工作记忆 (Working Memory)                │  ← LLM 上下文窗口
│  系统提示 + 当前对话 + 检索到的相关记忆 + 任务状态     │
├─────────────────────────────────────────────────────┤
│            情景记忆 (Episodic Memory)                 │  ← 向量数据库
│  每段交互的完整记录 + 时空元数据                       │
│  检索: recency × importance × relevance 三权重        │
├─────────────────────────────────────────────────────┤
│            语义记忆 (Semantic Memory)                 │  ← 知识图谱 / 关系型数据库存储
│  从情景中提炼的抽象知识: 用户画像、偏好、关系状态       │
│  定期反思生成，固化持久，主要记录用户画像               │
├─────────────────────────────────────────────────────┤
│            过程记忆 (Procedural Memory)               │  ← 系统提示模板
│  说话风格模板、行为决策树、情绪响应规则                 │
│  保证"像一个人"，而非每次随机生成。主要记录虚拟人本身的内容 │
├─────────────────────────────────────────────────────┤
│            遗忘管理器 (Forgetting Manager)             │  ← 生命周期策略
│  短期(x轮对话)完整保留 → 中期（短期记忆到达上限）总结压缩  │
│  + 永久(核心人格/关键事实)抽取输入用户画像            │
│  容量上限触发重要性剪枝                               │
├─────────────────────────────────────────────────────┤
│            冲突消解器 (Conflict Resolver)              │  ← 冲突消解
│  新事实写入时检测语义冲突                               │
│  标记旧记忆为 deprecated + 保留时序记录                 │
│  语义记忆更新为最新值，情景记忆保留原始对话              │
└─────────────────────────────────────────────────────┘

# 3.1 数据流：一次对话的完整生命周期

用户说话 → 同步路径（<100ms）:
  1. 向量检索情景记忆 (top-5)
  2. 加载语义记忆 (用户画像/关系状态/人格卡/约定)
  3. 拼装工作记忆 → LLM 生成回复
          → 异步触发（后台，不阻塞对话）:
  4. 写入消息队列 (Redis Stream)
  5. Worker1 消费:
     a. 关键事实提取 (轻量 LLM)
     b. embedding 生成
     c. 写入向量库
     d. 冲突检测与消解
  6. Worker2 消费:
     a. 关键信息提取 (轻量 LLM)（所提取的内容可能是我们之前枚举的用户画像、偏好、关系状态、约定等等）
     b. 冲突检测与消解
     c. 写入用户画像
  6. 定时任务 (每日凌晨):
     b. 遗忘: 按生命周期策略压缩/剪枝
     c. 关系更新: 计算好感度/信任度变化

# 3.2 冲突消解

这是现有方案中最常见的盲区。场景：用户 Day 1 说"我是程序员"，Day 5 说"我转行做设计了"。对于这种前后信息不一致问题的原因是原生 RAG 依赖语义相似度来匹配信息。但情景记忆的核心是“时间、顺序、因果、状态”。比如用户问：“我上周比这周买得多吗？”纯向量检索很难处理这种基于时间序列的对比查询，因为它只看词义距离，不理解时间逻辑；情景记忆是动态的。比如用户昨天说“我喜欢蓝色”，今天说“我现在更喜欢红色了”。纯向量库只会把这两句话都存下来，检索时可能同时召回，缺乏“更新/覆盖”旧记忆的机制

业内有成熟的解决方案

1，工程测在保存实体时增加时间戳或者版本号机制。同时给模型生成回复时，增加对应的 prompt，比如在 Prompt 中明确指示模型：“如果信息存在时序或矛盾，请永远使用最新的一条记忆。“

2，暴力枚举出记忆类型，比如用户性别、用户职业、用户关系等等重要信息，模型整理记忆时只会保留最新一条记忆。例如用 user_id + "address" 作为唯一键，每次新地址写入时物理替换旧地址。

当然你可能会问一个类型下可能有多个记忆怎么办，比如用户喜好这一类别，用户可以说自己之前喜欢吃火锅，今天说自己喜欢旅游，事实上用户都喜欢，因此不能简单的覆盖旧记忆。事实上我们可以定义哪些类型是需要覆盖的，哪些是不需要覆盖的。而覆盖与否可以交给模型判断

3，GraphRAG 基于知识图谱的检索增强生成可以处理相关问题，替代纯向量的黑盒相似性匹配。与只做语义匹配的向量 RAG 不同，GraphRAG 的核心区别在于将数据构建成一个显式的知识图。这种结构化方式，带来了处理复杂问题的关键能力，比如在存储时除了实体（如“我”），GraphRAG 还明确存储它们之间的关系（如“-曾住→ A地址”、“-现住→ B地址”）。当被问及地址，它可以沿着关系路径进行多跳推理，准确找到最新、最相关的地址节点。缺点是建设成本高、更新删除实体时消耗大

# 3.3 联系感的心理学基础

"人类的联系感是如何形成的"——心理学已有成熟答案

核心理论：相互依存记忆（Transactive Memory, Wegner 1985）

联系感不是"记住更多"，而是"我知道你知道，且你知道我知道"。一对老朋友之间的默契，不是因为他们记住了每一次对话，而是因为他们共享了一套知识体系——彼此知道对方知道什么

人类联系感要素 AI 记忆系统对应
共同经历回忆 情景记忆 + 主动提起
被理解感 语义记忆（用户画像/偏好）
关系进展感知 关系状态追踪
默契/共同知识 元记忆："我知道你记得什么"
脆弱性展示 agent 主动承认不确定

后两项是现有方案普遍缺失的。具体建议：

  1. 元记忆（shared_knowledge）：在语义记忆中增加字段，记录"agent 知道用户知道什么"和"用户知道 agent 知道什么"。当 agent 说"你上次提到的那个……"时，它不是在检索，而是在表达"我记得你知道这件事"

  2. 弃答策略：当 agent 不确定时，说"我不太确定你上次说的是不是这个，你还记得吗？"比编造一个答案好一百倍。这直接解决"胡言乱语"痛点，同时增强联系感——因为承认不确定本身就是一种亲密行为

记忆体系可以增加联系感的方法也就这么多，核心的设计还是在产品功能上去实现，比如增加用户的被需要感

# 3.4 反思机制的具体算法

参考斯坦福 Generative Agents (Park et al., 2023) 的三步反思：

Step 1: 重要性评分 — 对每条情景记忆打分 (1-10)
Step 2: 反思触发 — 当最近 100 条记忆的平均重要性 > 阈值时触发
Step 3: 反思生成 — LLM 回答 "关于 [主题]，以下 3 条观察能得出什么结论？"
         → 生成高层语义记忆

斯坦福小镇中为了让智能体快速做出反应，给记忆分区：

  • 容量小的瞬时记忆，行动前马上就能想到（根据 max token 去设置）

  • 容量大的长期记忆，需要回忆一下才能想起来的（很像人类）。 记忆会越攒越多，所以我们定期让智能体整理记忆（例如我可以设置在晚上睡觉的时候）方便检索。甚至可以加上很多拟人的设定：

  • 刻骨铭心的记忆没办法轻易忘掉

  • 太久远的记忆会因不断地整理而失真（约定太久，我记错了）

  • 即便智能体想要记住瞬时记忆中无聊的内容，也无法做到。（考前复习记不住知识点的人类）

# 3.5 人格一致性

每一轮对话注入一张人格约束卡：

你是阿雅，一个住在游戏小镇的冒险家助手。
- 说话风格：温暖、轻度幽默、偶尔用感叹词
- 情绪基调：乐观，但遇到危险时会严肃
- 知识边界：{{熟悉小镇的人和事，不知道游戏外的世界}}
- 与玩家的关系：{{信任度 72%，好感度 68%，上次一起完成了"矿洞探险"}}
